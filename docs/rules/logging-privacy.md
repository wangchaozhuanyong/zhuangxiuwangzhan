# 日志、审计和隐私规则

涉及后台、线索、报价、通知、媒体、管理员的操作，必须考虑日志和隐私。

## 日志规则

- 后台敏感操作应写入 audit log 或 system log。
- 删除、归档、权限变更、通知配置变更、批量修改都属于敏感操作。
- 日志要能帮助排查问题，但不能泄露密钥和隐私。
- 用户可见错误要用友好文案，不要直接展示数据库内部错误。
- 系统日志页面不能直接展示原始 `source`、`event_type`、`error_code`、英文 `message`，必须按 `docs/rules/user-facing-technical-fields.md` 映射成当前语言文案。
- 日志可以保存技术字段用于排查，但展示层必须区分“给开发者看的字段”和“给用户看的文案”。

## 隐私规则

- 日志里禁止输出 `SUPABASE_SERVICE_ROLE_KEY`、token、密码、cron secret、Webhook secret。
- 线索手机号、邮箱、地址、客户留言属于敏感信息，调试时不要随便打印完整内容。
- 备份文件、导出文件、测试日志要按敏感数据处理。
- 对外报错不能直接暴露数据库英文错误、表名、策略细节或密钥信息。

## 备份和导出

- 备份前确认目标环境和密钥来源。
- 备份结果不要上传到公开位置。
- 恢复前必须 dry run。
- 恢复生产前必须先在 staging 或测试项目验证。

## 禁止事项

- 禁止把密钥、token、密码写进日志或提交到仓库。
- 禁止为了调试打印完整客户隐私信息。
- 禁止把生产备份当普通静态文件处理。
- 禁止对外展示原始数据库错误。
- 禁止在后台日志、通知、错误提示中直出 `snake_case`、`kebab-case`、内部错误码或英文系统句子。
