# 用户界面禁止直出技术字段规则

这条规则用来防止页面出现“像乱码”的体验。它通常不是编码损坏，而是把后端、数据库、日志里的技术字段直接展示给了用户。

## 专业定义

以下情况统一按开发问题处理：

- 本地化不完整。
- 技术字段泄漏到用户界面。
- 原始后端日志或错误直出。
- `snake_case`、`kebab-case`、内部错误码、事件类型、状态码直接出现在中文或英文页面。

真正的编码乱码通常会出现 `�`、`锟斤拷`、`æ` 这类异常字符。技术字段直出虽然不是编码坏了，但同样属于不合格界面。

## 强制规则

- 用户能看到的页面、后台表格、弹窗、toast、错误提示、空状态、日志详情、通知消息，禁止直接展示后端原始技术字段。
- 数据库和 API 可以保存稳定代码，例如 `system_health_check`、`permission_denied`、`upload_failed`，但页面必须映射成人能看懂的文案。
- 中文页面不能直接出现未确认的英文系统句子，例如 `System health check passed.`。
- 中文页面不能直接出现裸露的 `snake_case` 或 `kebab-case` 技术值，例如 `system_health_check`、`health-check`。
- 英文页面也不能直接出现数据库表名、RLS 策略名、内部函数名、原始异常栈。
- 没有映射时，用户界面必须显示安全兜底文案，例如“未知系统事件”“未知状态”“操作失败，请稍后重试”。
- 缺失映射可以在开发日志或系统日志中记录，但不能把原始技术字段直接显示给普通用户或管理员。

## 必须映射的字段类型

以下字段一旦进入用户界面，必须先经过统一字典或 i18n 转换：

- `status`
- `state`
- `type`
- `event_type`
- `source`
- `severity`
- `level`
- `role`
- `permission`
- `error`
- `error_code`
- `message`
- `reason`
- `bucket`
- `provider`
- `channel`

如果字段本身就是品牌名、URL、API 名、SKU、文件名、邮箱、电话号码，可以保留原值，但必须确认它确实是业务内容，不是内部技术字段。

## 推荐实现方式

- 系统文案统一放在 `src/i18n`。
- CMS 内容走后台多语言字段。
- 状态、角色、日志来源、事件类型、错误码要有集中映射表，不要散落在页面里。
- 后端日志优先保存 `event_type`、`message_key`、`metadata`，不要只保存英文句子。
- 前端展示日志时，用 `event_type` 和 `metadata` 生成当前语言文案。
- 用户可见错误必须先经过 `src/lib` 中的错误转换或友好文案封装。

## 系统日志特别规则

系统日志允许保存技术字段，因为它们用于排查问题；但后台日志页面不能默认直出技术字段。

例如数据库可以保存：

```text
source = health-check
event_type = system_health_check
message = System health check passed.
```

页面必须展示为：

```text
来源：健康检查
类型：系统健康检查
信息：系统健康检查通过。
```

如果以后新增日志类型，必须同时补：

- 来源显示名。
- 事件类型显示名。
- 成功、失败、警告信息的用户友好文案。
- 中英文两套文案。
- 缺失映射时的安全兜底。

## 禁止写法

- 禁止把 `row.event_type`、`row.source`、`row.status` 直接渲染到页面。
- 禁止把 `error.message` 直接弹给用户。
- 禁止把 Supabase、数据库、RLS、Edge Function 的英文原始错误直接显示在 UI。
- 禁止为了让 i18n 检查通过，把真实需要翻译的技术直出加入 allowlist。
- 禁止只给中文补映射，不给英文补映射。

## 修改后检查

涉及用户可见文案、状态、错误、日志、通知时，必须检查：

- 页面是否还有裸露的 `snake_case`、`kebab-case`、英文系统句子。
- 中文和英文文案是否都补齐。
- 没有映射时是否显示安全兜底，而不是显示原始字段。
- 是否需要更新 `src/i18n`、状态字典或日志事件字典。
- 是否运行或说明未运行 `npm run i18n:check`。
- 是否运行或说明未运行 `npm run ui:text-check`。
