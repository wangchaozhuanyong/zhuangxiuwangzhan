# 表单和数据提交规则

所有表单都必须完整处理用户体验和服务端安全。联系表单、报价表单、后台表单都适用。

## 前端表单必须有

- 必填校验。
- 格式校验，例如电话、邮箱、URL。
- loading 状态。
- 成功提示。
- 失败提示。
- 防重复提交。
- 移动端可用布局。
- 用户失败后的兜底行动，例如 WhatsApp、电话或返回列表。

## 服务端或数据库必须有

- 服务端校验，不能只信前端。
- 写入权限控制。
- 防刷或限流，尤其联系表单、报价表单。
- 必要时记录提交来源、IP hash、时间、状态。
- 写入成功后再通知；通知失败不能抹掉已经保存的数据。

## 写入操作必须考虑

- 是否会重复提交。
- 是否需要幂等。
- 是否需要刷新 React Query cache。
- 是否需要记录 audit log 或 system log。
- 是否需要处理多人同时编辑冲突。
- 是否需要回滚或补偿更新。

## Flashcast 现有契约

- 联系和报价提交固定走 Supabase Edge Function `submit-lead`。
- 现有 `submit-lead` 返回格式 `{ ok, id }` / `{ error }` 先保持不动。
- 前端校验规则要和后端 `submit-lead` 校验保持一致。
- 后台保存内容后，必须刷新对应后台列表和前台 published cache。

## 禁止事项

- 禁止提交失败时显示成功。
- 禁止只在前端校验敏感规则。
- 禁止后台写入成功但列表、详情、前台仍显示旧数据。
- 禁止因为通知失败而丢掉已经保存的线索或报价。
